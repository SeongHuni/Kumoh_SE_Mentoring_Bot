# RAG 학과 멘토링 챗봇 설계

## 핵심 결정 3개

1. 대상 사용자: 신입생, 예비 신입생, 복학생, 편입생을 1순위로 두고 재학생을 2순위로 둔다.
2. 지식 범위: 학과 소개, 교육과정, 교수 소개, 졸업 요건, 장학금, 비교과 프로그램, 취업 현황, FAQ, 학사 공지, 학과 행사를 다룬다.
3. 문서 구조: 초기에는 주제별 Markdown 파일로 분리하고, 추후 DB에 지식을 저장하는 구조로 전환한다.

## 아키텍처

초기 버전은 Markdown 문서를 로더가 읽어 `KnowledgeDocument`로 변환한다. 추후 버전은 DB에 저장된 지식 레코드를 같은 `KnowledgeDocument` 형태로 변환해 검색 계층에 전달한다. 검색 계층은 문서를 chunk로 분할하고 Chroma에 인덱싱한 뒤 질문과 관련도 높은 chunk를 반환한다. 서비스 계층은 검색 결과를 바탕으로 출처가 포함된 답변 초안을 만든다. API 계층은 FastAPI로 `/health`, `/documents`, `/index/rebuild`, `/chat` 엔드포인트를 제공한다.

초기 버전은 운영 LLM 연결 전에도 동작하도록 로컬 해시 임베딩과 Chroma 기반 검색 결과로 답변 초안을 반환한다. 이후 OpenAI, Azure OpenAI, 또는 로컬 LLM을 연결할 때는 임베딩 함수와 `rag_service.py`의 답변 생성 부분을 교체한다.

## 저장소 전환 계획

Markdown 파일은 초기 구축과 사람이 읽기 쉬운 시드 데이터로 사용한다. 운영 단계에서는 DB 테이블에 지식을 저장하고, 관리자 화면이나 import 스크립트로 지식을 갱신한다.

권장 DB 구조는 다음과 같다.

- `knowledge_documents`: 문서 ID, 제목, 대상 사용자, 본문, 마지막 확인일, 소유자, 활성 여부
- `knowledge_sources`: 문서 ID, 출처 URL, 출처 이름, 확인일
- `knowledge_keywords`: 문서 ID, 검색 키워드
- `knowledge_chunks`: 문서 ID, chunk 번호, chunk 본문, embedding ID 또는 벡터DB 참조

이 구조를 쓰면 Markdown에서 DB로 옮겨도 API와 검색 서비스의 핵심 인터페이스를 크게 바꾸지 않아도 된다.

## 지식 문서 공통 필드

- 문서 ID
- 대상 사용자
- 요약
- 상세
- 출처 URL
- 마지막 확인일
- 관리 메모
- 검색 키워드

## 출처 정책

답변은 관련 문서 제목과 출처 URL을 함께 반환한다. 불확실한 질문은 추측하지 않고 확인이 필요한 항목으로 안내한다.

## 테스트 기준

문서 로더는 Markdown 메타데이터와 본문을 안정적으로 읽어야 한다. 챗 서비스는 질문에 대해 관련 문서와 출처를 반환해야 한다.
